Part II

Product Scope and Business Rules

Work2Cash is a unified-user, Nigeria-wide task marketplace. The product allows a user to hire help as a Task Owner and earn as a Tasker after KYC approval.

Persona

Unified User

One account can operate as Task Owner or Tasker. Tasker actions require approved TaskerProfile and KYC.

Payment

Escrow-first

Task funds are held before matching or direct offer acceptance. Payment state is confirmed by backend, not frontend redirect.

Discovery

Browse plus sort

Taskers can see available tasks sorted nearest first and filter by location. Tasks are not hidden strictly by radius.

MVP Scope

Area Included
Launch Coverage Nationwide Nigeria rollout. Task site must be in Nigeria. Tasker must be in Nigeria to accept. Task Owner can be anywhere.
Payment Providers Paystack and Moniepoint
MVP Modules Auth, KYC, Task Owner flow, Tasker flow, wallet/escrow, chat, voice messages, masked calls, notifications, admin operations, support, disputes/reports, withdrawals, referrals
Deferred Promo codes, subscriptions, loyalty, advanced analytics, in-app voice calls, multi-country support

Launch Task Categories

Errands Market runs; Supermarket shopping; Pharmacy runs; Picking up items; Items delivery; Fuel station runs

Task Discovery, Location and Privacy Rules

Rule Area Source-of-Truth Decision
Task visibility Tasks are not hidden strictly by radius. Taskers can browse available tasks sorted nearest first and farthest last.
Location filtering Taskers can filter available tasks by location. Radius is used for sorting, filtering, and cost control, not as the only visibility rule.
Nigeria boundary Task site must be in Nigeria. Tasker must be in Nigeria to accept a task. Task Owner can be anywhere.
Arrival time Task Owner sets required arrival time. Tasker must confirm they can arrive on or before that time.
ETA decision Task Owner sees Tasker ETA and decides whether to accept or reject the Tasker. The platform does not automatically reject by ETA in v1.
Address privacy Public task listing shows limited details. Exact address and full proof media are not publicly exposed.

Rule Decision

Tasker proof Tasker must upload proof of work done after completion/finalization.
Supported media Photos and videos.
Maximum file size 50MB per file.
Access control Media access must be permission-controlled and should use signed URLs or equivalent restricted access.
Admin moderation Admin can review and remove unsafe, invalid, unrelated, or policy-violating media. Removal reason must be logged.

Communication Rules

Channel Decision
Chat Task-bound chat is available after task acceptance.
Voice messages Supported as part of task communication and stored with permission-controlled access.
Masked phone calls MVP uses proxy dial-in flow. Real phone numbers must not be exposed.
Future in-app calls Deferred. Can be introduced after MVP if operationally justified.
Call availability Masked calls are task-bound and time-limited. Valid states include accepted, en route/in transit, arrived, in progress, or valid dispute window.
Off-platform leakage Before calls, show: For your protection, keep task communication and payment inside Work2Cash.

Pricing and Wallet Rules

Minimum Tasker Withdrawal ₦1,000

Cancellation, Penalty and Strike Rules

Scenario Rule
Task Owner cancels before Tasker accepts No penalty
Task Owner cancels after Tasker accepts No penalty
Task Owner cancels while Tasker is en route 10% penalty + 1 warning
Task Owner cancels after Tasker arrives 10% penalty + 1 warning
Tasker cancels after accepting 1 warning
Tasker no-shows 1 strike
Task Owner no-shows 10% penalty + 1 warning
Completion refused by Task Owner Tasker raises structured dispute/report; admin resolves
Warning conversion 3 warnings = 1 strike

Tasker Strike 1: cannot accept tasks for 2 days; Strike 2: 1 week; Strike 3: 1 month; Strike 4: account suspended
Task Owner Strike 1-3: trust score reduces; Strike 4: account suspended and must contact support

Part III

Actor Flows and Acceptance Criteria

This section is the product-flow source of truth. It explains how Task Owners, Taskers, and Admins move through Work2Cash, the system responsibilities at each stage, and the acceptance criteria QA must validate before go-live.

Cross-Actor Marketplace Flow

Architecture Diagram 
Task Owner                      Work2Cash Backend                         Tasker
    |                                  |                                     |
    | Create task + task site          |                                     |
    | Upload proof of work needed      |                                     |
    | Confirm geocoded map pin         |                                     |
    | Set required arrival time        |                                     |
    | Pay into escrow ---------------> | Verify payment + hold escrow        |
    |                                  | Publish limited task preview ------> | Browse tasks sorted nearest first
    |                                  |                                     | Filter by location/category/time
    |                                  |                                     | Confirm arrival ability
    | <------------------------------- | ETA + Tasker interest                |
    | Accept/reject Tasker ----------> | Lock accepted Tasker                 |
    |                                  | Enable chat, voice note, masked call |
    | Track arrival + communicate      | <--------- live tracking ----------> | Mark en route
    |                                  | <--------- location events --------> | Tap "I have arrived"
    |                                  |                                     | Tap "Begin Task"
    | Monitor task progress            |                                     | Upload completion proof
    | Confirm or report issue -------> | Release escrow or hold for review    |
    | Rate / favorite Tasker           | Update trust, wallet, history        | Await payout batch

The Work2Cash marketplace is escrow-led. The Task Owner funds the task first, the Tasker confirms arrival ability, and the Task Owner decides whether to accept the Tasker after seeing ETA. The backend remains the source of truth for task state, payment state, escrow, settlement, and policy enforcement.

Task Owner Flow

Task Owner Intent

The Task Owner flow is designed to make task posting clear, safe, and evidence-backed. A Task Owner can be anywhere, but the task site must be in Nigeria. The Task Owner must define the work, confirm the location, fund the task, review interested Taskers, and approve completion before final settlement.

Task Owner Flow Diagram

Architecture Diagram 
Register/Login
  -> Complete profile
  -> Create task from approved category
  -> Enter live/manual task address
  -> Geocode manual address when needed
  -> Confirm map pin
  -> Upload proof of work to be done
  -> Set required arrival time
  -> Review task amount + fees
  -> Pay into escrow
  -> Choose public discovery or direct favorite offer
  -> Review Tasker ETA and profile
  -> Accept or reject Tasker
  -> Communicate through chat, voice note, or masked call
  -> Track journey and task progress
  -> Review Tasker completion proof
  -> Confirm completion or raise task-linked report
  -> Rate Tasker
  -> Optionally add Tasker to favorites

Task Owner rejection is allowed before the Tasker starts moving and within 5 minutes after the Tasker marks en route. Rejections after movement are logged. Repeated movement-stage rejection can trigger warnings, trust score changes, or risk review.

Stage Explanation System Responsibility
Task Definition Task Owner selects an approved category/type, describes the work, provides address, confirms map pin, sets arrival requirement, and uploads proof media. Validate category, geocode address, enforce Nigeria task-site boundary, store proof media securely, and prevent public exposure of exact address/media.
Funding Task Owner reviews task amount, platform/service/payment fee, and final payable amount before payment. Initialize Paystack/Moniepoint payment, verify backend payment confirmation, create escrow, and persist provider references.
Tasker Selection Task Owner can use public discovery or direct favorite offer. Public discovery lets Taskers browse and express interest. Direct offer targets one favorite Tasker through API/FCM. Expose limited public task info, calculate ETA, protect exact address/media until acceptance rules are satisfied, and enforce rejection-window rules.
Communication After acceptance, Task Owner can chat, exchange voice messages, or call through Work2Cash masked proxy. Enable Socket.io chat/tracking, persist communication metadata, create TaskCallSession for masked calls, and prevent real phone number exposure.
Completion Task Owner reviews completion proof, confirms completion, rates Tasker, favorites Tasker, or raises a task-linked report. Release escrow to Tasker wallet or hold settlement if disputed, update ratings/favorites/trust, and audit sensitive outcomes.

Tasker Flow

Tasker Intent

The Tasker flow is designed around verified earning and clear status updates. A Tasker cannot accept tasks until KYC is approved. Once active, the Tasker can browse Nigerian tasks, confirm arrival ability, perform work, upload completion proof, and receive settlement through the scheduled payout process.

Tasker Flow Diagram

Architecture Diagram 
Register/Login
  -> Complete profile
  -> Submit KYC
  -> KYC approved
  -> Create/activate TaskerProfile
  -> Select skills/categories
  -> Set availability and travel radius
  -> Browse available tasks sorted nearest first
  -> Filter by location/category/time
  -> Open limited task preview
  -> Confirm ability to arrive by required time
  -> Accept public task or direct offer
  -> Tap "Start Journey" / mark en route
  -> Travel to task site with live tracking
  -> Tap "I have arrived"
  -> Wait for valid task state / Task Owner visibility
  -> Tap "Begin Task"
  -> Perform task
  -> Communicate through approved channels
  -> Upload proof of work done
  -> Request completion
  -> Receive settled earning in Tasker wallet
  -> Submit withdrawal request
  -> Receive payout in bulk batch on 14th or 28th

The Tasker must be in Nigeria to accept tasks. Tasker status updates are explicit button-driven transitions: Start Journey moves the task to en route, I have arrived moves the task to arrived, and Begin Task starts the in-progress work state. These transitions must be persisted and validated by the backend.

Stage Explanation System Responsibility
Activation Tasker completes profile, KYC, skills/categories, availability, travel radius, and location permission. Block task acceptance until KYC is approved and TaskerProfile is active.
Discovery Tasker sees available tasks sorted nearest first and can filter by location rather than being limited to radius-hidden visibility. Return limited task previews, protect private address/proof media, and enforce Nigeria tasker-location acceptance rule.
Commitment Tasker confirms they can arrive on or before the Task Owner's required arrival time. Calculate/display ETA, track Tasker confirmation, and prevent acceptance outside Nigeria.
Journey Tasker taps Start Journey/en route, travels to the task site, and taps "I have arrived" on arrival. Persist TASKER_EN_ROUTE and ARRIVED states, stream live tracking, apply ETA guard, and expose journey status to Task Owner.
Execution Tasker taps "Begin Task" to move from arrival to active task execution. Persist IN_PROGRESS state, enable task progress UI, enforce valid state transitions, and keep communication available only in valid states.
Settlement Tasker uploads completion proof and receives earning after Task Owner confirmation or admin resolution. Release escrow to Tasker wallet, apply commission tier, hold only disputed/active-issue earnings, and include eligible withdrawal requests in the next payout batch.

Admin Operations Flow

Admin Intent

The admin dashboard is the operational control center for trust, finance, support, KYC, coverage, configuration, and platform health. Every high-risk action must be permission-controlled, require a reason where appropriate, and create an audit trail.

Admin Operations Flow Diagram

Architecture Diagram 
Admin login
  -> TOTP verification
  -> RBAC permission check
  -> Dashboard overview
      +-- Users and sessions
      +-- Tasker KYC/review
      +-- Active/historical tasks
      +-- Payments, escrow, withdrawal requests
      +-- Payout batches for 14th and 28th
      +-- Reports/disputes and risk flags
      +-- Support live chat
      +-- Task media review
      +-- Categories and pricing/config
      +-- Coverage controls
      +-- Use case health and background jobs
      +-- Audit logs and monitoring
  -> Action requires reason if high-risk
  -> System writes AdminAuditLog

Admin operations must never bypass finance, KYC, security, or dispute rules arbitrarily. Payout batches require finance review, settlement eligibility checks, and audit logging before bulk transfer execution.

Admin Area Explanation Controls
KYC Review Tasker identity verification and manual-review cases. KYC Reviewer role, restricted identity data, audit logged decisions.
Finance Monitor payments, escrow, wallet ledger, refunds, withdrawal requests, payout batches, and reconciliation. Finance Admin role, reason-required actions, ledger consistency checks, payout calendar control.
Payout Batch Prepare bulk payout on the 14th and 28th for eligible Taskers who submitted withdrawal requests. Eligibility check, dispute hold check, provider transfer file/API execution, success/failure reconciliation, and audit log.
Reports/Disputes Resolve structured task-linked reports with evidence; disputes do not use live chat. Support/Risk roles, evidence review, resolution audit.
Support Live Chat Assist users through operational live support separate from dispute adjudication. Support Agent role, session assignment, message persistence.
Media Moderation Review and remove unsafe or invalid task proof media. Removal reason, media status update, risk flag for repeated abuse.
Coverage/Config Control nationwide service availability by state, city, or custom geo-zone. Operations/Super Admin role, change reason, audit log.

Product Flow Acceptance Criteria

Flow Launch Acceptance Criteria
Task Owner Can create, geocode, pin-confirm, fund, match/direct-offer, accept/reject Tasker, communicate, confirm, rate, favorite, and report/support.
Tasker Cannot accept before KYC approval; can browse location-filtered tasks, confirm arrival ability, accept/decline, tap Start Journey, tap "I have arrived", tap "Begin Task", communicate, upload proof, receive earning, and request withdrawal.
Admin TOTP/RBAC enforced; can manage users, Taskers, KYC, tasks, finance, reports/disputes, support live chat, categories, coverage config, audit logs, use case health, withdrawal requests, and payout batches.
Payout Eligible Tasker withdrawal requests can be batched and paid on the 14th and 28th, while disputed/active-issue Tasker payouts require admin authorization.
Privacy Exact address and full proof media are not public. Real phone numbers are not exposed during task-bound calls.
Cost Control Maps ETA is batched/guarded. Provider costs are logged and monitored.

Task Discovery State Rule

Updated Discovery Model

Taskers can view available tasks sorted nearest first and farthest last, with location filters. Radius is used for ranking, filters, and cost optimization, not as a hard visibility wall.

Lifecycle and Data Integrity Rules

Area Rule
Task state Task transitions must be explicit, validated, and persisted. Invalid state jumps must be rejected.
Task media Media belongs to a task and must have uploader, role, purpose, status, file metadata, permission scope, and moderation fields.
Geocoding Manual addresses must produce coordinates and a pin confirmation record before posting proceeds.
Wallet ledger Balances must never change without an append-only WalletTransaction record.
Escrow Escrow release/refund/hold must be tied to task state, completion, dispute/report outcome, or admin-authorized workflow.
Settlement hold Only disputed or active-issue tasks delay payout. Non-disputed Tasker earnings should settle normally into the Tasker wallet.
Payout batching Withdrawal requests are collected into payout batches for the 14th and 28th of every month. A PayoutBatchItem links each eligible Withdrawal to the batch, provider reference, transfer status, and reconciliation outcome.
Warnings/strikes Policy events must be role-specific. The same user can have separate Task Owner and Tasker policy histories.
Audit High-risk admin, finance, KYC, policy, media-removal, and configuration actions must create audit records.

Not Socket-Based

API Response and Error Standard

Contract Area Standard
Success shape { success: true, message, data, meta }
Error shape { success: false, message, error: { code, details } }
Pagination List endpoints must expose pagination metadata in meta, including page/cursor, limit, total where practical, and next cursor where cursor-based.
Idempotency Payment, webhook, wallet, media upload finalization, task state transitions, payout batches, and settlement endpoints must support idempotency where duplicate calls can occur.
Correlation REST, socket, worker, webhook, and payout flows should carry requestId/correlationId for logs and use case health tracking.
Sensitive data Responses must not expose KYC secrets, real phone numbers, private addresses before acceptance, provider secrets, OTPs, or hidden media URLs.

Socket Contract Rules

Rule Decision
Authentication Socket connection must authenticate with a valid user token and bind to user/session context.
Authorization Joining task rooms requires valid task membership and task state.
Persistence Task chat, support live chat, voice message metadata, and delivery/read receipts are persisted asynchronously where needed.
Reconnection Mobile must refresh task state from REST after reconnect because socket events are not durable source of truth.
Scaling When multiple API/socket instances exist, Socket.io must use Valkey/Redis adapter and proxy/load-balancer rules must support stable connections.

Engineering Standards

Engineering rules define how the Backend, Mobile, and Admin Frontend teams keep the codebase maintainable, observable, testable, and aligned with product decisions. These standards are binding for implementation unless formally changed in this architecture document.

Backend Standards

  • NestJS with Hexagonal Architecture.
  • Every use case has explicit input params, response shape, domain errors, and instrumentation.
  • Track use case name, status, durationMs, lastUsedAt, requestId, actor where safe, error code, module, and environment.
  • Do not log sensitive KYC, payment, OTP, phone, address, media, or provider secret data.
  • Use adapters for Paystack, Moniepoint, Smile ID, Google Maps, Termii, FCM, storage, and masked call providers.
  • Use idempotency for payment, wallet, webhook, media, payout batch, and critical task state transitions.
  • Backend must publish the APIs needed by Mobile and Admin before the end of Week 1 using contract/stub responses where real logic is not ready.
  • Target file size below 400 lines; hard warning above 500 lines.

Mobile Standards

  • Flutter feature-first clean architecture.
  • Explicit use cases, params, entities, models, repositories, datasources, and state classes in separate files.
  • Riverpod with autogenerated classes preferred.
  • Every feature must define idle, loading, success, empty, error, offline, retrying, and permission denied states where relevant.
  • Location, camera, microphone, media, notification, and phone permissions must have product-specific justification and graceful denial handling.
  • Tasker task state buttons must map clearly: Start Journey = TASKER_EN_ROUTE, "I have arrived" = ARRIVED, "Begin Task" = IN_PROGRESS.

Admin Frontend Standards

  • Next.js feature-first architecture with typed API clients and permission-aware routes.
  • Every admin table supports loading, empty, error, retry, filters, pagination, and permission denied states.
  • Finance, KYC, policy, support, payout batch, and config actions must request reason where high-risk.
  • Audit log visibility must be built into high-risk workflows.

Frontend Loading and Submission States

State Type Required UX Standard
Initial fetch / GET request Show a skeletonized version of the target view. Cards should show skeleton cards, tables should show skeleton rows, and detail pages should show skeleton sections instead of generic spinners.
Refresh / background fetch Keep current content visible where safe and show subtle refreshing indicators. Do not blank out the screen unless data is unavailable.
POST/PATCH/critical submit Show a blur overlay above the current view with a spinning Work2Cash logo. This applies to task posting, payment init, profile save, KYC submit, proof upload finalization, withdrawal request, payout approval, and high-risk admin actions.
DELETE/destructive action Require confirmation, show blur overlay with spinning Work2Cash logo during submission, and show success/error confirmation after completion.
Upload progress Show file-level progress for photos/videos and support retry for failed files. Max file size is 50MB per file.
Payment pending Show pending state and backend status refresh. Never mark payment successful based only on frontend redirect or user tap.

Commit Discipline

No Stacked Feature Commits

Commits must be made feature by feature and fix by fix. One commit should represent one clear, reviewable change. Do not bundle two unrelated features, a feature plus a bug fix, or backend/mobile/admin changes that are not part of the same logical unit.

  • Good: feat(tasks): add tasker arrived endpoint
  • Good: fix(wallet): prevent duplicate payout batch item
  • Good: test(tasks): cover begin task transition
  • Bad: feat(app): add task flow, wallet payout, admin dashboard and fixes
  • Money, KYC, admin permission, security, provider, payout, and migration commits must include a body explaining the behavior and risk.

Session behavior Unlock state is stored in sessionStorage, so a new browser session requires password entry again.
Direct document guard Protected HTML pages inside documents/ must include <script src="../assets/js/auth-guard.js"></script>.
Return path The guard preserves the originally requested URL with a next query parameter and redirects the user after successful password entry.
Security limitation The GitHub Pages guard is a soft/team-convenience control only because the site and repository are public.
Forbidden content No secrets, API keys, production credentials, private user data, sensitive payment data, confidential operational secrets, or production records should be stored in the repo.
