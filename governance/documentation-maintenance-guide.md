# Work2Cash Documentation Maintenance Guide

## Purpose

This guide keeps the completed Phase 0–6 documentation structure readable,
focused and reproducible. It is for maintainers, reviewers and AI agents making
documentation changes after the migration.

## Normal reading surface

The portal exposes six current libraries:

1. Document Registry.
2. Standalone Flow Library.
3. Junior Developer Plans.
4. Contract Decision Library.
5. Technical Reference Library.
6. Assurance and Governance.

Legacy and superseded artifacts remain available through the registry as
migration evidence. Do not restore them to normal portal navigation.

## Before editing

1. Check `governance/document-registry.md`.
2. Identify the canonical source and its authority.
3. Confirm the change owner and required reviewers.
4. Record a conflict or gap instead of guessing.
5. Edit one documentation family at a time.

## Generated families

| Change | Edit | Regenerate |
| --- | --- | --- |
| Portal routes, glossary or library cards | `content/portal/portal.json` | `node scripts/generate-portal.js` |
| Flow behavior or explanation | `content/flows/` | `node scripts/generate-flow-docs.js` and the flow inventory |
| Accepted contract decision | `content/contracts/` | `node scripts/generate-contract-docs.js` |
| Weekly execution work | `content/tasks/<team>/week-*.md` | Task and execution-plan generators |
| Technical record | `content/technical/` | `node scripts/generate-technical-docs.js` |
| QA, legal or governance record | `content/assurance/` | `node scripts/generate-assurance-docs.js` |
| Registry metadata | `governance/document-registry.json` | `node scripts/generate-document-registry.js` |

The portal search index, AI Agent Start Here and Full Project Agent Context are
generated by `scripts/generate-portal.js`.

## Document-count control

Create a new human page only when all of the following are true:

- The material has a distinct audience and task.
- It cannot remain an anchored record in an existing family.
- A canonical source, owner, reviewers and generator are defined.
- Portal navigation remains simpler after the addition.

Do not create one page per endpoint, model, provider, screen, test, legal clause,
decision, risk or daily build task. Add those as records inside their existing
focused family.

## Agent-context control

- Use focused flow, weekly, technical and assurance packs for implementation.
- Use one team-week pack, not standalone daily prompt files.
- Use Full Project Agent Context only for onboarding, architecture review or
  cross-document consistency work.
- Treat generated context as non-canonical.

## Required validation

Run before handoff:

```text
node scripts/generate-portal.js --check
node scripts/generate-document-registry.js --check
node scripts/validate-docs.js
node scripts/validate-accessibility.js
node scripts/apply.guard.js
git diff --check
```

Also run the generator check for every changed documentation family.

## Review cadence

- Review active and approved registry entries at least quarterly.
- Review a family immediately after an authority, product rule, contract,
  provider or release-policy change.
- Keep provisional gaps visible until the named owner supplies evidence.
- Record approvals, reversals and migration changes in
  `governance/review-log.md`.
